pl-pipeline 是一套通用、语言无关、IDE 无关的 AI 研发流水线框架。 用三层架构（piao / pl / adapter）把"需求→实现→验证→归档"的全链路做成 机器可执行的 DSL，让每一个 gate 都有可追溯的 trace 事件。

为什么做这件事

AI 编码时代的三个致命痛点

01

门禁是散文，不是判据

"编译通过 + lint 通过 + 单测通过" —— 每句话都对，但没有脚本真的在判定。 人类说"过了"它就过了。AI 说"过了"它也过了。gate.eval 是口头承诺。

"看起来绿色的流水线，真跑 demo 才发现 7 个 bug。"
02

验证只到编译，不到启动

静态检查再全，也抓不到"装上依赖启动 500"、"module state 被 HMR 清空"、 "passlib × bcrypt 5.x 坏组合"这类只有真启动才能暴露的问题。

VERIFY 只管静态，OBSERVE 只管线上，中间缺 cold-start smoke。
03

规则是 AI prompt，不是 linter

"revalidateTag 对 module-state 数据源无效" —— 这条知识在 skill 里写得清清楚楚，但没有任何脚本在 CI 扫代码看谁违反。 知识只服务 AI 生成，不服务 CI 验收。

一个漏网 bug 只会被人工修一次，下次还会再犯。
核心架构

三层分工 · 职责清晰 · 互不侵蚀

piao-kernel 管"事实归档（语义底座）"、pl-core 管"流水线执行"、 adapters 管"场景填充"。每一层只暴露契约，内部实现可独立演化。

关键突破

纸面契约 → 可执行契约

所有 AI 编码工具的隐秘弱点：评估判据不是机器可执行的。 v1.1 把它修了。

Before · v1.0

纸面门禁

gates:
  D:
    criteria:
      - "lint PASS"
      - "compile PASS（./gradlew build）"
      - "网络层自检 PASS"
      - "解析层自检 PASS"
  • 散文判据，没有脚本真的求值
  • 硬编码宿主栈（gradle / ApiSelfCheck）
  • gate.eval 和 check.run 无闭环
  • lint 失败，gate 仍可写 passed
After · v1.1

可执行契约

gates:
  D:
    checks: [compile_check, lint, test]
    eval: "all_checks.pass"
    on_failure: block

checks:
  compile_check:
    cmd: "$PL_BUILD_CHECK_CMD"
    timeout_sec: 180
  lint: { cmd: "$PL_LINT_CMD" }
  test: { cmd: "$PL_TEST_CMD" }
  • 结构化判据，pl-runner.sh 机器求值
  • adapter 注入命令，pl-core 零耦合
  • gate.eval 自动派生自 check.run
  • lint 失败 → gate 真的 blocked

pl-pipeline 自我演化飞轮

v1.1 的所有能力增强，都不是事先规划出来的 —— 而是在自己用自己的过程中跑出来的。

1
纸面 demo 全绿
两个 reference adapter 都通过 schema/validate，文档显示 "ARCHIVE 11/11"。

2

↓
3
retro v1：归咎 adapter
开了 6 个方案，4 个都是"让 adapter 多声明字段"。——被用户一句点破：不要被 adapter 困住。
↓
4
retro v2：回到本体视角
4 条本体缺陷：E1 可执行契约层 / E2 SMOKE 阶段 / E3 rule-as-code / E4 契约-现实漂移。 判断标准：能力如果所有 adapter 都要各做一遍，它属于 pl/piao 本体。
↓
5
v1.1 落地 3/4
E1 + E2 + orchestrator 解耦（pl-v1.1.0-alpha）· E4 contract-drift（pl-v1.1.1-alpha）。 剩 E3 待 v0.3。
↓
6
反证实验闭环
回滚 Next.js demo 到 retro-v1 修之前 → 新的 piao-contract-drift 当场抓到 6 个 error，覆盖 B1/B4/B6/B7 四个原始 bug。能力真的在了。

元洞见

真正的"做了一次 → 系统变强"飞轮，不靠计划，靠 adversarial test： 用流水线做真事情，发现漏洞归因到本体缺陷，把修补变成通用契约，下次自动受益。

30 秒上手

把 pl-pipeline 装进你的项目

1

克隆独立仓

git clone https://github.com/walkertop/pl-pipeline-standalone.git
export PL_HOME=$PWD/pl-pipeline-standalone
2

装 adapter 到你的项目

cd ~/your-nextjs-project
bash $PL_HOME/scripts/adapter-install.sh \
     $PL_HOME/adapters/adapter-nextjs-web .
3

创建第一个 change

# 把 pl-core 的 config 拷过来
cp $PL_HOME/assets/pl/config.default.yaml pl/config.yaml

# 手动 or AI 生成 5 件产物
mkdir -p pl/changes/my-first-feature
# spec.md / plan.md / taskdag.md / api.md / .state.md
4

跑 D 门禁 + SMOKE + drift

PL_PROJECT=$PWD bash $PL_HOME/scripts/pl-runner.sh \
  --change my-first-feature --gate D

PL_PROJECT=$PWD bash $PL_HOME/scripts/pl-smoke.sh \
  --change my-first-feature

PL_PROJECT=$PWD bash $PL_HOME/scripts/piao-contract-drift-compute.sh \
  --change my-first-feature
