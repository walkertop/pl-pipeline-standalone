## 2. 为什么要升级

### 2.1 v1.4 时代的痛点

在 v1.4 及之前，`assets/pl/` 下只有：
- `config.default.yaml`
- `observe-rules.default.yaml`
- `templates/`

**问题**：新用户 `cp` 配置、执行 `/pl:proposal` 时发现：命令是空壳——底层调用的 `spec-normalizer` skill 并不在独立仓里，物理上还住在某个源项目（KuiklyPolyCity）中。

**根因**：v1.4 完成了"流水线骨架 + 观察层 + Dashboard"，但没来得及把"通用能力"从孵化源项目搬出来。

### 2.2 v1.5 的解法

**三层分层落地**：

```
┌──────────────────────────────────────────┐
│ Project Layer                            │ ← 用户项目
│   <user-project>/.codebuddy/             │
└──────────────────────────────────────────┘
                  ↑ 覆盖
┌──────────────────────────────────────────┐
│ Stack Layer (Adapters)                   │ ← v1.5 起完整
│   adapters/adapter-<stack>/              │
│   例：adapter-kotlin / adapter-nextjs-web│
└──────────────────────────────────────────┘
                  ↑ 覆盖
┌──────────────────────────────────────────┐
│ Generic Layer (Core Assets)              │ ← v1.5 起完整
│   assets/pl/{skills,rules,agents}/       │
└──────────────────────────────────────────┘
```

## 3. 升级步骤（已接入 pl-pipeline 的项目）

### 3.1 拉取新版本

```bash
cd /path/to/pl-pipeline-standalone
git fetch --tags
git checkout pl-v1.5.0-rc.1       # 或 pl-v1.5.0（正式版发布后）
```

### 3.2 用户项目侧：三层加载策略

用户项目的 `.codebuddy/` 无需改动。pl-core 的加载顺序：

```
(1) <user-project>/.codebuddy/<asset>/    ← 最先读
(2) adapters/adapter-<stack>/<asset>/     ← 其次读（通过 .pl-adapter.yaml 识别 stack）
(3) assets/pl/<asset>/                    ← 兜底
```

**已有项目怎么办？**
- 如果你的 `.codebuddy/` 下已有这些资产（例如从 KuiklyPolyCity 直接复制）：
  - 它们自动成为项目级 override，继续生效
  - **建议 diff 对比**（你的版本 vs `assets/pl/` 版本），合并通用改进
- 如果你的项目没有这些资产：
  - 自动使用独立仓的通用版本，无需任何操作

### 3.3 想叠加栈级规范？安装对应 adapter

例：Kotlin 项目想获得 `kotlin-code-review` skill。

```bash
cd /path/to/your-kotlin-project

bash $PL_HOME/scripts/adapter-install.sh \
  $PL_HOME/adapters/adapter-kotlin \
  .
```

完成后：

```
.pl-adapter.yaml                            ← 自动生成，记录注入元数据
.codebuddy/rules/kotlin-coding-standards.md
.codebuddy/rules/kotlinx-serialization.md
.codebuddy/skills/kotlin-code-review/SKILL.md
scripts/adapter-kotlin-{build,verify,lint}.sh
```

## 5. Breaking 与兼容性

### 5.1 Non-Breaking 变更 ✅

- `pl/config.yaml` schema 未变
- `pl/changes/<id>/` 产物契约（7 件）未变
- `.state.md` 门禁字段未变
- `scripts/pl-*.sh` 的 JSON schema 未变
- 已有 change 的归档文件无需重生成

### 5.2 Minor Breaking ⚠

1. **`build-verification` rule 的 gradle 命令外化**
   - 原 rule 直接写 `./gradlew compileKotlin`
   - 新 rule 只写 `通过 .pl-adapter.yaml 的 build_adapter.commands 读取`
   - **影响**：若项目之前直接复制 rule 且不经 adapter 装配，需建立 `.pl-adapter.yaml`（或安装对应 adapter）

2. **文件名归一化**
   - `git-commit-rules.md` → `git-commit.md`
   - `serialization-rules.md` → `kotlinx-serialization.md`（更准确反映是 kotlinx.serialization 规范）
   - `acceptance-criteria-rules.md` → `acceptance-criteria.md`
   - **影响**：如有文档/脚本直接引用旧文件名，需更新

3. **`pipeline-master` agent 六阶段 → 七阶段**
   - 新增 `SMOKE` 阶段（介于 `VERIFY` 和 `OBSERVE` 之间）
   - **影响**：自定义的 `.state.md` 若硬编码阶段枚举需兼容新阶段

### 5.3 迁移脚本（若你需要自动处理）

```bash
# 若你的项目中有硬编码的旧文件名引用，用此脚本扫描
grep -rE 'git-commit-rules\.md|serialization-rules\.md|acceptance-criteria-rules\.md' \
  --include="*.md" --include="*.sh" --include="*.yaml" \
  your-project/
```

## 8. 常见问题

### Q1: 我的项目同时装了 adapter-kotlin 和 adapter-android，规则冲突了怎么办？

A: `adapter-android` 应在 `adapter.yaml` 中声明 `requires: [adapter-kotlin@>=1.0.0]`，并在自己的 rule 中使用 **更具体的 Rule-ID**（如 `ANDROID-001-LIFECYCLE`）避免与 `KT-001-*` 冲突。加载器会保留两层规则，不同 ID 不冲突。

### Q2: KuiklyPolyCity 项目内的资产还需要保留吗？

A: 看你希望这些资产作为"项目 override"还是"通用默认"。建议：
- **保留你对通用版本的定制部分**（例：自定义的业务 scope 枚举）
- **删除纯拷贝**（与独立仓完全一致的部分）
- 通过 `.codebuddy/` 的 override 机制只保留**差异化**内容

### Q3: `spec-normalizer` 迁出后，KuiklyPolyCity 执行 `/pl:proposal` 还能用吗？

A: 能。三层加载策略会自动退化：
- 若 `.codebuddy/skills/spec-normalizer/` 仍存在 → 用项目版
- 若不存在 → 用 `assets/pl/skills/spec-normalizer/`

建议：删掉项目内的 spec-normalizer 拷贝，diff 一下看是否有你需要保留的改动。

### Q4: 我想贡献自己的 adapter（例如 adapter-go / adapter-rust），该怎么做？

A: 读 [`docs/guides/adapter-authoring.md`](./guides/adapter-authoring.md)，以 `adapter-nextjs-web` / `adapter-kotlin` 为参考实现照着写，通过 `adapter-validate.sh` 后开 PR。

### Q5: v1.5 后的路线图？

A:
- v1.6: `adapter-android` / `adapter-kmp-kuikly`（叠加在 adapter-kotlin 之上）
- v1.7: `adapter-go` / `adapter-rust`
- v1.x: `adapter-common`（跨语言通用工具：pre-commit hook / changelog 生成器等）

完整 roadmap 见 `ROADMAP.md`。
