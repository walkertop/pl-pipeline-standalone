# v1.13 · Trace Schema 全面切 OpenTelemetry（规划）

> **写于**：2026-04-25（pl-v1.12.0 完成、shop_agent 首例集成期间）
> **目标版本**：pl-v1.13.0（breaking change，schema 重大版本）
> **状态**：📋 规划中（**未启动**），等 shop_agent PoC 反馈后开工
> **作者动机**：pl 当前的私有 trace JSONL 在工程事件层做得很扎实，但
> (a) 缺 agent / model / token 维度，无法覆盖未来 pl-master 多 agent 编排；
> (b) 完全私有 schema 与社区 LLM 可观测生态割裂，无法直接接 Langfuse / Phoenix / Datadog；
> (c) 想加新维度时全靠自己定义，没有标准词表可参照。
>
> 本文论证："**全切 OpenTelemetry GenAI + 通用 SemConv 扁平 JSONL**"是否能完整覆盖
> 现有功能、能带来什么额外收益，以及如何落地。

---

## 0. TL;DR（最终决策）

- **采用 (A2) OTel-shaped 扁平 JSONL**：1 行 = 1 span，字段名严格按 OTel 标准（`trace_id` / `span_id` / `parent_span_id` / `name` / `attributes` / `events`），LLM/Agent 调用按 GenAI SemConv，pipeline 业务字段用 `pl.*` 私有命名空间。
- **配套一键导出器** `pl trace export-otlp`：把扁平 JSONL 转标准 OTLP/JSON batch，可直接 POST 给 Langfuse / Phoenix / Tempo / Datadog 等任意 OTel 后端。
- **完全替换** v1.12 的私有 schema（不双轨并存），所有消费方（`trace-report.sh` / `pl-trace-tree.sh` / retro-mine / observe-rules / dashboard）一并重写。
- **不做迁移器**：v1.12 之前的 trace 归档冻结按旧 schema 存留；v1.13 起新写入。
- **预计工作量**：~3-4 个工作日（含 PoC 验证 + 上游迁移 + 文档）。

## 1. 现状（v1.12 trace 完整画像）

### 1.1 Schema

```json
{
  "ts": "2026-04-22T12:33:40Z",
  "trace_id": "<change-id>_<YYYYMMDD>_<HHMMSS>",
  "change_id": "<change-id>",
  "phase": "SPEC | PLAN | IMPLEMENT | VERIFY | SMOKE | OBSERVE | ARCHIVE",
  "actor": "script:<name>@<version>",
  "event": "<event-type>",
  "span_id": "sp_<nano>_<rand>",          // v1.6.1+
  "parent_span_id": "sp_<nano>_<rand>",   // v1.6.1+
  "data": { /* 任意嵌套 JSON */ }
}
```

### 1.2 emit API 全集（来自 `scripts/trace-emit.sh`）

| API | 写出的 event | 是否开 span | 用途 |
|---|---|---|---|
| `trace_phase_start` / `_end` | `phase.start` / `phase.end` | ✅ push/pop | Pipeline 阶段生命周期 |
| `trace_gate_start` / `_end` | `gate.start` / `gate.eval` | ✅ push/pop | 门控评估 |
| `trace_task` start/end | `task.start` / `task.end` | ✅ push/pop | TaskDAG 任务 |
| `trace_artifact` create/update | `artifact.create` / `artifact.update` | ❌ point | 文件创建/修改 |
| `trace_check` | `check.run` | ❌ point | lint / tsc / test 等检查 |
| `trace_adapter_use` | `adapter.use` | ❌ point | Consumer→Adapter 能力消费证据（CDC） |
| `trace_runtime_log` | `runtime.log` | ❌ point | 运行时日志（MigrationLogger 桥接） |
| `trace_error` | `error` | ❌ point | 错误事件 |
| `trace_init` / `<workflow>` | `workflow.start` / `workflow.end` | ✅ push/pop | 顶层 workflow span（v1.6.1+） |

### 1.3 消费方

- `scripts/trace-report.sh` — 出 markdown 报告
- `scripts/pl-trace-tree.sh` — 重建 span 树
- `scripts/trace-adapter-use.sh` — 抽 CDC 证据
- `assets/piao/scripts/retro-mine.sh` — 模式挖掘
- `assets/pl/scripts/observe-rules.sh` — 规则违反检测
- Dashboard live-reload UI

### 1.4 缺什么

- ❌ **agent 维度**：`actor` 字段同时承载脚本和 agent，无标准化拆分
- ❌ **model / provider 维度**：不知道某次操作走了哪个 LLM
- ❌ **token / cost 维度**：无法做成本核算
- ❌ **tool 调用维度**：agent 调用了什么 tool（Read/Bash/...）没记录
- ❌ **跨工具生态**：私有 schema，Langfuse/Phoenix/Datadog 等都不认
- ❌ **未来 pl-master 多 agent 编排**：现有 schema 不足以表达"哪个 agent 在哪个 task 里调用了哪些 tool/agent"

## 2. 方案对比

### 2.1 三个候选

| 方案 | 描述 |
|---|---|
| **(0) 维持现状 + 加字段** | 保留 v1.12 schema，往 `data` 里加 `agent_name`、`model`、`tokens`，自定义新事件类型 |
| **(A1) 严格 OTLP/JSON batch** | 完全切 OTel wire format，1 行 = 1 batch（`resourceSpans → scopeSpans → spans`），后端直吞 |
| **(A2) OTel-shaped 扁平 JSONL** ⭐ | 1 行 = 1 span，字段名按 OTel 标准 + `gen_ai.*` + `pl.*`，配 jq 转换器一键变 OTLP |

### 2.2 维度对比

| 维度 | (0) 现状+ | (A1) 严格 OTLP | (A2) 扁平 JSONL ⭐ |
|---|---|---|---|
| 字段名标准化 | ❌ 私有 | ✅ 完全 OTel | ✅ 完全 OTel |
| 后端直吞（Langfuse/Phoenix） | ❌ 写适配器 | ✅ 直接 POST | ⚠️ 经一个 jq 转换 |
| `grep` / `tail -f` 友好 | ✅ | ❌ 嵌套深 | ✅ |
| `jq` 单行查询 | ✅ | ❌ 钻嵌套 | ✅ |
| 老脚本改动 | 🟡 小 | 🔴 全重写 | 🔴 全重写 |
| schema 演进 | 🔴 自维护 | ✅ 跟 OTel | ✅ 跟 OTel |
| 学习成本 | 0 | 🔴 高（OTLP 协议） | 🟡 中（OTel 字段名） |
| 未来导入 OTel 后端 | 🔴 写适配器 | ✅ 已经是 | ✅ 一行命令 |
| AgentOps / LangSmith 生态 | ❌ | ✅ | ✅ |
| 离线纯 JSONL 操作 | ✅ | ⚠️ 难读 | ✅ |
| 跟 Cursor/Claude Code 未来 OTel 输出对接 | ❌ | ✅ | ✅ |

### 2.3 为什么不是 (0)

- 短期容易，长期债务无穷：每加一个维度都要自己定义 schema、写 schema 文档、写消费方迁移
- **无法接入任何成熟 LLM 可观测生态**——所有可视化 / 评估 / 成本看板都得自己造
- 把 pl 锁在自己的小池子里，与 2024-2026 行业标准化趋势相反

### 2.4 为什么不是 (A1)

- 严格 OTLP/JSON 是**批传输**协议，不是"日志格式"——一行装一批 spans（嵌套 `resourceSpans → scopeSpans → spans`），单行不可读
- bash + jq + tail -f 工具链全废
- pl 99% 场景是**本地、离线、文件即数据**，强上 OTLP 等于把"日志"伪装成"网络协议"，得不偿失
- **真正需要 OTLP 时再转就行**，不需要常态以 OTLP 为存储格式

### 2.5 为什么是 (A2)

- **字段名标准 = 语义标准**：`gen_ai.usage.input_tokens` 任何 LLM 工具都认，pl 不再造词
- **行结构保留 = 工具链保留**：`jq` / `tail -f` / `awk` / `grep` 全部沿用
- **一键变 OTLP = 生态打通**：~30 行 jq 转换器就能 POST 给 Langfuse/Phoenix/Datadog
- **自定义有处可放**：`pl.*` 命名空间是 OTel 官方鼓励的"vendor namespace"做法
- **本质是给 pl trace 做"语义规范化"**，不是引入新依赖

## 3. 选定方案细节（v1.13 schema）

### 3.1 单行 span 完整结构

```json
{
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id":  "00f067aa0ba902b7",
  "parent_span_id": "b7ad6b7169203331",
  "name": "<verb> <subject>",
  "kind": "INTERNAL | CLIENT | SERVER | PRODUCER | CONSUMER",
  "start_time_unix_nano": 1714123456789000000,
  "end_time_unix_nano":   1714123459123000000,
  "status": {
    "code": "OK | ERROR | UNSET",
    "message": "<可选错误描述>"
  },
  "attributes": {
    "service.name":       "pl-pipeline",
    "service.version":    "1.13.0",
    "pl.change_id":       "<change-id>",
    "pl.phase":           "IMPLEMENT",
    "pl.actor.kind":      "script | agent | orchestrator | user",
    "pl.actor.name":      "pl-runner | sqlmodel-author | ...",
    "pl.actor.version":   "1.0.0",
    "pl.trace.alias":     "<change-id>_20260425_011200",   /* 保留人类可读 ID */
    "...其他业务字段...":  "..."
  },
  "events": [
    {
      "name": "pl.<event_name>",
      "time_unix_nano": 1714123457000000000,
      "attributes": { /* 扁平 KV */ }
    }
  ]
}
```

### 3.2 Span / SpanEvent / Log 三态划分

| 现 v1.12 事件 | v1.13 表达 | 说明 |
|---|---|---|
| `workflow.start` / `workflow.end` | **Span**（root，name=`pipeline.workflow`，attrs=`pl.change_id`） | 一次 change 的总时长 |
| `phase.start` / `phase.end` | **Span**（child of workflow，name=`pipeline.phase IMPLEMENT`） | 阶段时长 |
| `gate.start` / `gate.eval` | **Span**（child of phase，name=`pipeline.gate D`，attrs=`pl.gate.expr`/`pl.gate.result`） | 门控评估有耗时 |
| `task.start` / `task.end` | **Span**（child of phase，name=`pipeline.task T012`） | 任务时长 |
| `check.run` | **Span**（child of task or gate，name=`check tsc`，status 表达 pass/fail） | 检查项执行 |
| `artifact.create` / `update` | **SpanEvent** on enclosing span（name=`pl.artifact.created`，attrs=`pl.artifact.path` / `pl.artifact.sha256`） | 瞬时事件 |
| `adapter.use` | **SpanEvent**（name=`pl.adapter.used`，attrs=`pl.adapter.id` / `pl.adapter.version` / `pl.asset.kind` / `pl.asset.id`） | CDC 证据 |
| `error` | Span 的 `status.code = ERROR` + SpanEvent `name = "exception"`（attrs=`exception.type` / `exception.message` / `exception.stacktrace` —— OTel 标准） | 错误处理 |
| `runtime.log` | **OTel Log signal**（独立日志流，通过 trace_id/span_id 关联） | 运行时日志 |
| **新增**：`agent.invoke` | **Span**（name=`invoke_agent <name>`，attrs=GenAI SemConv） | pl-master 子 agent 调用 |
| **新增**：`llm.chat` | **Span**（name=`chat <model>`，attrs=GenAI SemConv） | LLM 直接调用 |
| **新增**：`tool.execute` | **Span**（name=`execute_tool <tool>`，attrs=GenAI SemConv） | Read/Bash/Edit 等 tool 调用 |

### 3.3 GenAI SemConv 字段（agent / llm / tool span 必须遵守）

```jsonc
{
  "gen_ai.operation.name":      "invoke_agent | chat | execute_tool | embeddings",
  "gen_ai.provider.name":       "anthropic | openai | aws.bedrock | ...",
  "gen_ai.request.model":       "claude-sonnet-4.5",
  "gen_ai.response.id":         "msg_xxx",
  "gen_ai.response.finish_reasons": ["stop"],
  "gen_ai.usage.input_tokens":  12450,
  "gen_ai.usage.output_tokens":  1850,

  "gen_ai.agent.name":          "sqlmodel-table-author",
  "gen_ai.agent.id":            "pl/agents/sqlmodel-author.md@sha256:abc",

  "gen_ai.tool.name":           "Read",
  "gen_ai.tool.call.id":        "call_xxx",

  // opt-in（默认关闭，敏感）
  "gen_ai.system_instructions": "...",
  "gen_ai.input.messages":      "[{...}]",   // JSON.stringify
  "gen_ai.output.messages":     "[{...}]"
}
```

### 3.4 `pl.*` 私有命名空间（自定义业务字段全集）

| 前缀 | 用途 | 示例字段 |
|---|---|---|
| `pl.change_id` | 变更标识 | `db-readonly-bootstrap` |
| `pl.phase` | 七阶段 | `IMPLEMENT` |
| `pl.actor.*` | 谁在做（kind/name/version） | `script` / `pl-runner` / `1.12.0` |
| `pl.trace.alias` | 人类可读 trace_id | `<change>_<ymd>_<hms>` |
| `pl.task.*` | TaskDAG 任务（id/name/user_story） | `T012` |
| `pl.gate.*` | 门控（id/expr/result） | `D` / `all_checks.pass` / `passed` |
| `pl.check.*` | 检查（name/cmd/status/duration_ms） | `tsc` |
| `pl.artifact.*` | 产物（path/sha256_before/sha256_after） | — |
| `pl.adapter.*` | adapter（id/version） | `python-fastapi` / `1.7.0` |
| `pl.asset.*` | 资产消费（kind/id） | `skill` / `db-async-only` |
| `pl.rules_loaded` | array<string> 本 span 加载了哪些 rule | `["db-async-only"]` |
| `pl.skills_loaded` | array<string> 本 span 加载了哪些 skill | — |
| `pl.user_story` | 用户故事关联 | `US1` |
| `pl.cdc.*` | Consumer-Driven Contract 字段 | — |

## 4. 功能保留性论证（逐 API 映射）

**关键问题**：v1.12 所有 trace_* API 在 v1.13 都能等价表达吗？逐项核对：

| v1.12 API | v1.13 等价实现 | 信息保留 | 备注 |
|---|---|---|---|
| `trace_phase_start "PLAN"` | `trace_span_start --name="pipeline.phase PLAN" --attr "pl.phase=PLAN"` | ✅ 完全 | 保持栈式 push |
| `trace_phase_end "pass"` | `trace_span_end --status=OK` | ✅ 完全 | 自动 pop |
| `trace_gate_start "D"` | `trace_span_start --name="pipeline.gate D" --attr "pl.gate=D"` | ✅ 完全 | — |
| `trace_gate_end "D" "passed"` | `trace_span_end --status=OK --attr "pl.gate.result=passed"` | ✅ 完全 | — |
| `trace_task start T012` | `trace_span_start --name="pipeline.task T012" --attr "pl.task_id=T012"` | ✅ 完全 | — |
| `trace_artifact create path` | `trace_event --name="pl.artifact.created" --attr "pl.artifact.path=path"` | ✅ 完全 | 写为 SpanEvent，挂在当前栈顶 span 上 |
| `trace_check tsc pass` | 长检查：`trace_span_start --name="check tsc"` + `_end --status=OK`<br>瞬时：`trace_event --name="pl.check.run" --attr "pl.check.status=pass"` | ✅ 完全 | **改进**：有耗时的检查从 point 升级为 span，可看 P95 |
| `trace_adapter_use skill db-async-only` | `trace_event --name="pl.adapter.used" --attr "pl.adapter.id=python-fastapi" --attr "pl.asset.kind=skill"` | ✅ 完全 | — |
| `trace_runtime_log domain INFO msg` | `trace_log --severity=INFO --body="msg" --attr "pl.log.domain=domain"` | ✅ 完全 | **改进**：升级为独立 OTel Log signal，与 trace 分离更干净 |
| `trace_error msg` | 当前栈顶 span 设 `status.code=ERROR` + `trace_event --name="exception" --attr "exception.message=msg"` | ✅ 完全 | **改进**：用 OTel 标准异常约定，APM 自动识别 |

**逐项核对完毕**：v1.12 所有 12 个 emit API 在 v1.13 都有等价表达，且其中 3 项有质量改进（`check.run` 升 span、`runtime.log` 升 Log signal、`error` 走标准异常约定）。

**⚠️ 一个真实损失：嵌套 `data` 必须扁平化**

- v1.12：`data: {nested: {key: val}}` 任意嵌套
- v1.13：`attributes` 只能是扁平 KV，值类型限 `string | int | bool | double | array<以上>`
- **影响**：retro-mine 等地方少量嵌套数据需要拍平为 `pl.foo.bar.baz=val`，或塞 `pl.detail.json="<JSON.stringify>"` 兜底
- **评估**：现有 `data` 大多是 1-2 层浅嵌套，扁平化后**反而更利于跨 span 聚合查询**（"所有 lint 平均耗时" 等）

## 5. 额外收益（OTel 才有，pl 现在没有）

切过去之后白送：

### 5.1 现成的语义约定（无需自己设计）
- **异常**：`exception.type` / `exception.message` / `exception.stacktrace` —— APM 直接识别
- **HTTP**：`http.method` / `http.status_code` —— 接 webhook 时直接用
- **DB**：`db.system` / `db.statement` —— migration 类操作直接用
- **Code**：`code.function` / `code.filepath` / `code.lineno` —— 错误定位
- **Cloud**：`cloud.provider` / `cloud.region` —— 部署相关

### 5.2 Resource 抽象（per-process 共享头）
现 v1.12 每行都重复 `actor`、`change_id` —— v1.13 用 `service.name` / `service.version` / `pl.host.os` 在 trace 启动时一次性写入，成为整 trace 共享元数据。导出 OTLP 时自动放 `resourceSpans.resource`。

### 5.3 Span Links（跨 trace 关联）
- "本次 change 引用了上次的 archive 文档" → `links: [{trace_id, span_id, attributes: {pl.relation: "references"}}]`
- "重试 trace 关联到原始 trace" → 同上
- 现 v1.12 完全做不到

### 5.4 Sampling / Batching（未来上规模）
- 上 OTel SDK 后内置降采样（如"check.run 全采，runtime.log 1/10"）
- batching 写入降低 IO（高频 emit 场景）

### 5.5 跨工具生态对接
- **Cursor / Claude Code 未来 OTel 输出**（Anthropic roadmap 已列）→ 可与 pl trace 拼接成完整链路
- **Datadog / Honeycomb / New Relic** → 团队场景可一行命令接入企业 APM
- **Jaeger / Tempo** → 本地开发可用 Docker compose 起 UI 看 span 树

### 5.6 标准 token / cost 计费
- `gen_ai.usage.input_tokens` / `output_tokens` → 任何 LLM 看板自动算钱
- pl 当前没有这维度，未来加 pl-master 时这是核心需求

### 5.7 Multi-agent 因果可视化
- pl-master → 子 agent → tool 调用，整树自然成形（现 v1.12 没有 agent 概念，无法表达）
- 这是切换的**第一原始驱动**（也是上一轮你提到的"哪个 agent 用什么 rules/skills 处理什么任务"的实现基础）

### 6.3 推荐组合（按 pl 阶段）

| 你的目标 | 推荐组合 | 启动时间 |
|---|---|---|
| **5 分钟看效果** | Phoenix（local SQLite）+ pl-otel-export.sh | 5 分钟 |
| **生产严肃自托管** | Langfuse + pl-otel-export.sh | 半天 |
| **重点观测 Cursor / Claude Code 子 agent** | AgentProbe（被动）+ pl 主动 emit | 1 天 |

| **agent 决策回放** | AgentOps SaaS | 1 小时 |
| **企业 OTel 平台对接** | 任何 OTel 后端（直接 POST） | 看公司 IT |

### 6.4 一个反直觉洞察

针对 pl 这种"prompt-driven、IDE 内 subagent 编排"的场景，**主流 SDK 仪器方案普遍失效**——因为：

- Cursor / Claude Code 跑子 agent 时，是**它们的进程**在调 LLM API，**pl 代码没机会拦截**
- OpenLLMetry / OpenInference 都需要在调用 LLM 的进程里 init
- 所以传统 SDK 路线**只能 instrument pl 自己的 SDK-side agent**（如果将来用 LangGraph），**给不了 Cursor 子 agent 打 trace**

→ **只有两条路能解决 Cursor/Claude Code 子 agent 观测**：

1. **被动 transcript 分析**（AgentProbe 路线）— 读 IDE 留下的对话日志
2. **prompt-level 自报告** — agent 通过 meta-rule / hook 自己写 trace 事件

→ 印证了 v1.13 的核心设计判断：**`agents.otel.jsonl` 的写入主体应是 pl 脚本/agent 自己**，不是 SDK 拦截。AgentProbe 是有趣的 hybrid，可作"被动验证层"补充，但不替代主动 emit。

## 7. 扩展能力（v1.13 之后能做什么）

### 7.1 一键起本地可视化
```bash
pl trace serve --backend phoenix     # docker compose 起 Phoenix
pl trace push                        # 把当前 change trace 推到本地 Phoenix
open http://localhost:6006           # 看 span 树 / token 成本 / agent 调用图
```

### 7.2 本地 jq 风格查询（仍然可用）
```bash
# 这次 change 用了哪些 agent
jq -r 'select(.attributes."gen_ai.agent.name") | .attributes."gen_ai.agent.name"' trace.jsonl | sort -u

# 总 token 成本
jq -s 'map(.attributes."gen_ai.usage.input_tokens" // 0) | add' trace.jsonl

# task → agent 多对多关系
jq -r 'select(.attributes."pl.task_id" and .attributes."gen_ai.agent.name") |
       [.attributes."pl.task_id", .attributes."gen_ai.agent.name"] | @tsv' trace.jsonl | sort -u
```

### 7.3 跨 trace 关联
- 标 `pl.relation: "retried-from"` + `links: [{trace_id, span_id}]` 表达"重试链"
- 标 `pl.relation: "references-archive"` 关联到上次 archive 的 trace_id
- retro-mine 可以跨 change 找模式

### 7.4 跟未来 Cursor / Claude Code OTel 输出拼接
- IDE 出的 OTel 直接 push 到同一个 Phoenix
- pl trace 与 IDE trace 共享 trace_id（约定 IDE side trace_id = `pl.trace.alias`）→ 一棵完整树
- 看到"用户对话→Cursor 调 Claude→Claude 调 pl→pl 调 sub-agent→sub-agent 调 tool"的完整链路

### 7.5 token 成本看板
- Langfuse / Phoenix 都内置 cost calculator
- 团队场景：周报"本周 pl 在 shop_agent 项目花了 $X，最贵的 task 是 Y"

### 7.6 agent 行为回放
- AgentOps / Phoenix 都支持 trace replay
- pl-master 调 sub-agent 失败时，能"回放"看每一步决策

### 7.7 Evals 与 A/B 测试
- Langfuse / Phoenix 都内置 dataset + eval
- "同一 task 用 agent A 和 agent B，比较成功率/token/质量"
- 配合 v1.12 的 retro-mine，做"哪些 rule 组合下 task 成功率高"的归因

### 7.8 与 piao 知识沉淀闭环
- piao kernel events 也按 OTel emit（用 `pl.piao.*` 命名空间）
- ARCHIVE 时把 trace 作为"事实账本"sealed 进 piao
- 长期形成"trace → 模式 → 资产 → 新 trace"的飞轮
