## 0. v1.5 后的状态复盘

v1.5 完成后我们拥有：

- ✅ `assets/pl/` 完整的通用资产（2 skills + 4 rules + 2 agents）
- ✅ 3 个 adapter（nextjs-web / python-fastapi / kotlin）
- ✅ Dashboard live reload + retro-miner + observe 层
- ✅ 完整的 pipeline 七阶段 + 七门禁
- ✅ 脱敏指南 + adapter-authoring 指南 + 辩证方法论指南

但**还没拿到过一份"完整的真实使用证据"**——所有验证都是工具链自验，没走过"新用户 clone → 新项目 → 全链路跑通 → 产真实 trace"的完整路径。

这就是下一步要解决的问题。

## 2. Path E'：真实使用验证 · **推荐优先**

### 做什么
跑**一个与 KuiklyPolyCity 无关的干净新项目**，完整经过 `/pl:proposal → plan → implement → verify → archive` 六步，验证：
- 核心资产（spec-normalizer / pipeline-master 等）是否真的能独立驱动流水线
- Dashboard live reload 是否在真实 trace 下健壮
- adapter 在新项目里 install 是否顺滑
- 三层覆盖机制在有项目 override 时是否按预期工作

### 场景候选（三选一）

| 候选 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **A. 新建一个 next.js todo app** | 最干净；已有 `adapter-nextjs-web`；能验证整条链路；能复用 `examples/demo-nextjs-todo` | 需要一点真实代码工作量 | ⭐⭐⭐⭐⭐ |
| B. 给 pl-pipeline 自己加小功能 | 工具链现成；能吃自己的狗粮 | 自举式验证说服力弱；业务特征不典型 | ⭐⭐⭐ |
| C. 新建 markdown parser CLI | 极干净；能验证 generic rule | 没有现成 adapter，需同时做 adapter-python 或 adapter-rust | ⭐⭐ |

**强烈推荐 A**：从零建一个小的 next.js todo app（或者直接把 `examples/demo-nextjs-todo` 升级到真实用法），**一次性完整走完流水线**。

### 产物清单

| 产物 | 位置 |
|------|------|
| 新建 demo 项目（或升级现有） | `examples/v1.5-real-run/` |
| 完整 trace | `pipeline-output/trace/<change>.events.jsonl` |
| Dashboard 截图 | `docs/retros/v1.5-real-run/dashboard-*.png` |
| 跑通 retro | `docs/retros/v1.5-real-run/retro.md`（发现的 bug、改进点） |
| 若发现核心资产 bug | 直接补丁回 `assets/pl/`（飞轮生效） |

### 关键观察点（写入 retro）

1. **spec-normalizer 足够通用吗？** 给非 Kuikly 项目出 spec，有没有 AI 不知所措的场景？
2. **pipeline-master 的七阶段是否自然？** SMOKE 阶段对小项目是不是冗余？
3. **build-verification 脱去 gradle 命令后，adapter-nextjs-web 的 `npm run build` 是否能自然替代？**
4. **Dashboard live reload 在真实产事件时会不会卡/丢？**（snapshot-lost bug 是否彻底修复）
5. **三层覆盖**：我把 rule 复制到项目 `.codebuddy/rules/` 改两行，pl-core 是不是真的用我的版本？

### 投入估算
- 准备 demo 项目：0.5 天
- 跑完整流水线 + 记录：0.5 天
- 写 retro + 修补发现的问题：0.5~1 天
- **合计 1.5~2 天**

## 3. Path F：扩充 adapter 生态

### 做什么
基于 adapter-kotlin 的基础层，做 2~3 个特化栈 adapter：

```
adapter-kotlin (已有，基础层)
    ├── adapter-android          (叠加 Android 特有规则)
    └── adapter-kmp-kuikly       (叠加 Kuikly KMP 特有规则)

adapter-go                       (新基础层)
```

### 每个 adapter 的产物
- `adapter.yaml`（含 `requires: [adapter-kotlin@>=1.0.0]` 或独立）
- `rules/` 特有规则（Android: Lifecycle / ViewBinding；Kuikly: Card/Module；Go: go-vet / errcheck）
- `skills/` 特有能力（Android: compose-review；Kuikly: kuikly-migration-checker；Go: goroutine-leak-detector）
- `scripts/{build,verify,lint}.sh`

### 价值
- **跨栈成立证明**：证明"栈级 adapter"机制不是孤证，能承载多个栈
- **KuiklyPolyCity 可回收**：adapter-kmp-kuikly 做完后，KuiklyPolyCity 可以**真正**变成"装 adapter" 的用户项目

### 投入估算
- adapter-android：1.5 天（有社区文档参考）
- adapter-kmp-kuikly：2 天（需与 KuiklyPolyCity 联调）
- adapter-go：1 天（生态成熟，规则明确）
- **合计 4~5 天**（可按需挑 1~2 个）

### 依赖
- E' 先做：E' 会暴露"adapter 机制本身的缺口"，修完再扩充才不会多份 adapter 同时踩坑

## 4. Path G：v0.2 CLI 打包

### 做什么
把 bash script 包一层 Node/Python CLI，降低"装 pl-pipeline"的门槛：

```bash
# 当前（v1.5）
cd my-project
export PL_HOME=/path/to/pl-pipeline-standalone
export PL_ASSETS=$PL_HOME/assets
cp $PL_HOME/assets/pl/config.default.yaml pl/config.yaml
mkdir -p .codebuddy/{agents,skills,rules} scripts pl/changes
bash $PL_HOME/scripts/adapter-install.sh $PL_HOME/adapters/adapter-nextjs-web .

# v0.2 愿景
npm install -g pl-pipeline        # or pipx install pl-pipeline
cd my-project
pl init                           # 自动完成以上所有步骤
pl install-adapter nextjs-web     # 自动下载 + 安装 adapter
pl doctor                         # 检查环境
```

### 产物
- `pl-pipeline` npm package（thin wrapper，内部仍是 bash scripts）
- `pl init` / `pl install-adapter` / `pl doctor` / `pl status` / `pl advance` 命令
- Adapter registry（先用 GitHub raw URL，v1.0 再做托管）

### 价值
- **降低使用门槛**：30 秒上手
- **建立分发标准**：adapter 有了版本号 + 独立下载渠道

### 投入估算
- CLI skeleton：1 天
- 4 个核心命令：2 天
- Adapter registry（MVP 版）：1 天
- 测试 + 文档：1 天
- **合计 5~7 天**

### 依赖
- E' 暴露的使用路径是 CLI 设计的依据（`pl init` 到底要干什么步骤，E' 里走一遍就知道了）

### 风险
- 跨平台打包（macOS / Linux / Windows）复杂度
- bash → Node/Python 桥接在 Windows 上可能踩坑

## 5. Path H：retro-miner B→C→D 链路

### 做什么
激活 v1.4 做的 retro-miner，让它吃真实 trace：

```
E' 产出的真实 trace (.events.jsonl)
    ↓
[B] retro-miner 跑挖掘（4 类 pattern：频次/共现/反模式/异常）
    ↓
[C] Dashboard 展示挖掘结果
    ↓
[D] E5 trace-verifier 契约 v0.1（用 B 积累的 pattern 反推契约规则）
```

### 产物
- `docs/retros/v1.5-real-run/retro-miner-output.md`（B）
- Dashboard 新增 "Patterns" tab（C）
- `pl/schemas/trace-verifier-v0.1.schema.json`（D）
- `assets/pl/rules/trace-contract.md`（D 落地）

### 投入估算
- B：0.5 天（跑 + 看结果）
- C：1 天（Dashboard UI）
- D：1~1.5 天（契约定义 + 样本验证）
- **合计 2~3 天**

### 依赖
- **强依赖 E'**：没有真实 trace retro-miner 无饭吃
- 弱依赖 F：如果多栈 adapter 产 trace，挖掘质量会更高

## 6. 推荐执行序列

根据「**依赖关系 + ROI + 风险**」三维度综合：

### 方案 A · 保守路径（推荐）

```
1. E' (1.5 天)          ← 先验证 v1.5 真的能用，出真实 trace
       ↓
2. H  (2~3 天)          ← 趁热打铁跑 retro-miner，闭合"观察-挖掘-反馈"
       ↓
3. F  (按需 1~5 天)     ← 根据 E' 反馈拍 adapter 投入范围
       ↓
4. G  (5~7 天)          ← 最后做 CLI（CLI 设计需要前三步积累的使用经验）
```

**总周期**：10~15 天 | **覆盖**：E' + H + 1~2 个 adapter + CLI MVP

### 方案 B · 快进路径（若用户想先有可分发物）

```
1. E' (1.5 天)          ← 同上
       ↓
2. G  (5~7 天)          ← 优先 CLI，尽快让"装 pl-pipeline"门槛降下来
       ↓
3. F + H（延后）         ← v1.7 再做
```

**总周期**：7~9 天 | **风险**：CLI 设计可能因 adapter 生态未成熟而返工

### 方案 C · 探索路径（若想先做生态规模）

```
1. F  (4~5 天)          ← 先把 3 个 adapter 做完看"机制是否真的可复用"
       ↓
2. E' (1.5 天)          ← 用多栈 adapter 跑真实新项目
       ↓
3. H + G                ← 后做
```

**风险**：F 可能在设计上犯和 adapter-kotlin 同样的错（没有真实项目反馈做不出真正通用的抽象）

## 7. 推荐：方案 A + 具体起点

### 第一步：E' 的场景拍板

提问给用户：
- **E'-A** 新建 next.js todo app（推荐⭐⭐⭐⭐⭐）
- E'-B 自举（给 pl-pipeline 自己加个 `pl doctor` 命令，合并进 G 早期）
- E'-C 新建 markdown-parser CLI（需要新 adapter，推荐度最低）

### 第二步：根据 E' retro 决定 F / H 的具体内容

E' 跑完后会出一份 retro，其中的"已识别问题列表"直接成为：
- F 的 adapter 特化优先级
- H 的 retro-miner 首批 pattern 目标
- 若有 core 资产 bug，直接出 `pl-v1.5.1` 补丁

### 第三步：G 在 H 之后做
- CLI 的最大价值是"复制已知好的流程"，所以必须先有"已知好的流程"（E' 提供了）
- CLI 需要的 adapter registry 格式由 F 的实际 adapter 数量 / shape 决定

## 附录 · 所有路径的并发可行性

这 4 条路径**互不排斥**，可以组合：

| | E' | F | G | H |
|--|--|--|--|--|
| E' | — | 可并行 | 需先做 | 需先做 |
| F | 可并行 | — | 可并行 | 可并行 |
| G | 需先做 | 可并行 | — | 弱依赖 |
| H | 需先做 | 可并行 | 弱依赖 | — |

若要**并行推进**，最稳的组合是：**E' 做完后，同时开 F + H**（二者互不干扰）；G 等前三者积累经验后做。
