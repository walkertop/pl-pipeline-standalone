# v1.14 · 门控系统演进（规划 · 调研）

> **写于**：2026-04-25（紧随 v1.13 trace OTel 规划）
> **目标版本**：pl-v1.14.0（新增能力，主要为非破坏性扩展）
> **状态**：📋 规划中（**未启动**）；本文先做**社区调研 + 范式分类 + 落地建议**，待 v1.13 trace
> 重构稳定后再开工。
> **作者动机**：v1.12 完成后，pl 的 trace（观测）做得扎实但门控（决策）系统单薄。
> 当前 8 个 gate（A0/B1/C/D/E/E_smoke/F/G）+ `eval: all_checks.pass` 三态枚举属于
> "流程阶段门控"范式，是社区八大门控范式之一。其余 7 大范式 pl 几乎空白：
> 没有 hook 拦截、没有 fail→re-generate 闭环、没有 sandbox、没有 AI critic、
> 没有结构化 approval。
>
> 本文论证 pl 应在哪些范式上补强、为什么、以什么 ROI 顺序、要避开什么坑。
> **不会盲目堆门控**——研究界 2024-2025 的实测教训：门控密度 ≠ 质量。

---

## 0. TL;DR

- **保留并强化** ⑦ **流程阶段门控**（pl 的强项，已是头部水平）。
- **优先补** ③ **测试反馈闭环**（方向 A，半天工作量，ROI 最高）：
  D/E gate 失败时把 stderr + 失败检查上下文喂回 agent，循环 N 次再判 block。
- **次优补** ④ **PreToolUse hook**（方向 B，1-2 天）：
  借 Claude Code / Cursor hook 机制，能拦危险 Bash、强制人确认 schema 改动。
- **再补** ⑤ **轻量 AI critic**（方向 C，1 天）：
  在 D pass 之后插一个 `pl-critic` agent，输出评论 + 阻断/通过决定。
- **能力升级** D **DSL 化的 `eval` 表达式**（方向 D，1 天）：
  替代当前 `all_checks.pass` 枚举，支持组合表达式 + 条件路由。
- **谨慎打开** E **Sandbox 模式**（方向 E，opt-in，2-3 天）：
  Docker 沙盒跑 check，环境隔离 + 多机可复现。
- **可选** F **结构化 approval token**（方向 F，0.5 天）：
  A0/B1 隐式签字升级为 `/pl:approve <gate>` 命令 + trace 审计事件。
- **不做**：完整重构成 MetaGPT/CrewAI 式多 agent SOP。研究界已证明这条路 ROI 不如预期。

## 1. 现状（v1.12 门控完整画像）

### 1.1 阶段与门控

| Stage | Gate (出口) | Type | 评估方式 |
|---|---|---|---|
| SPEC | A0 | external | `spec_complete` check |
| PLAN | B1 | external | `taskdag_valid` check |
| IMPLEMENT | C (内部) → D | internal/external | C: 所有 task DONE；D: `compile && lint` |
| VERIFY | E | external | `test` |
| SMOKE | E_smoke | external | smoke probes（adapter 注入）|
| OBSERVE | F (内部) → G | internal/external | F: 无 P0 异常；G: 知识沉淀完整 |
| ARCHIVE | (终态) | — | — |

### 1.2 评估机制

- **Schema**：`assets/pl/schemas/gate.schema.json` 定义 gate / check 数据模型
- **Eval**：枚举三选一 —— `all_checks.pass | any_checks.pass | majority_checks.pass`
- **on_failure**：枚举三选一 —— `block | warn | skip`
- **Check cmd**：支持 `${PL_BUILD_CHECK_CMD}` 等 adapter 注入
- **Evidence**：所有 check 结果落 trace JSONL，gate eval 读 trace

### 1.3 范式归类

按下面第 2 节的"八大范式"对照，pl 当前覆盖：

| 范式 | pl 覆盖度 |
|---|---|
| ① 静态检查门控 | ⭐⭐⭐⭐ |
| ② 执行/沙盒门控 | ⭐⭐ (有执行，无沙盒) |
| ③ 测试即门控 | ⭐⭐⭐ (有测试，无 retry 闭环) |
| ④ Hook/拦截门控 | ⭐ |
| ⑤ AI Critic 门控 | ⭐ |
| ⑥ 多 Agent SOP 门控 | ⭐⭐ (配置上有 owner_actor，实操是同一 LLM) |
| ⑦ 流程阶段门控 | ⭐⭐⭐⭐⭐ |
| ⑧ 人在环门控 | ⭐⭐ (隐式签字) |

## 2. 社区八大门控范式（每范式附代表项目 + 真实代码示例）

### ① 静态检查门控（最古老、最普及）

**机制**：拿到 patch/commit 后，跑 lint / type-check / SAST / SCA，失败即拒。

**代表**：pre-commit framework、husky + lint-staged、CodeQL、Dependabot、Snyk、Semgrep、SonarQube、mypy/pyright/tsc/ruff。

**与 LLM 关系**：完全解耦——LLM 写代码，静态检查独立判定。

**示例 1：pre-commit hook 配置**
```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: https://github.com/returntocorp/semgrep
    rev: v1.85.0
    hooks:
      - id: semgrep
        args: ["--config=auto", "--error"]
```

**示例 2：GitHub Actions 作为 PR 必过门控**
```yaml
# .github/workflows/quality-gate.yml
name: Quality Gate
on: [pull_request]
jobs:
  static:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install ruff mypy
      - run: ruff check .
      - run: mypy .
      - run: semgrep --config=auto --error
# 配合 GitHub branch protection rule 设为 required
```

**优点**：稳定、可解释、零幻觉
**局限**：只抓**语法层错误**，抓不到逻辑错误

---

### ② 执行/沙盒门控（"跑过才算数"）

**机制**：在隔离环境（Docker / VM / cloud sandbox）真实运行生成的代码，按运行结果判定。

**代表**：
- **Devin (Cognition)** — 云端 sandbox，agent 在里面跑代码 + 看错误 + 迭代
- **OpenHands** (前 OpenDevin) — 开源对标 Devin，提供 Docker sandbox + agent SDK
- **SWE-agent** — Princeton/CMU，Agent-Computer Interface (ACI) + sandbox，SWE-bench 跑分基础
- **OpenAI Codex CLI** — 本地 sandbox 模式
- **E2B** — sandbox 即服务（给所有 agent 框架用）

**示例：SWE-agent 工作循环（伪代码）**
```python
# SWE-agent 简化版
agent = SWEAgent(model="claude-sonnet-4.5")
sandbox = DockerSandbox(repo="org/repo", commit="abc123")

for step in range(max_steps):
    obs = sandbox.observe()       # 当前文件树 / 上次命令输出
    action = agent.decide(obs)    # ACI 限定动作集（ls/cat/edit/test）
    result = sandbox.execute(action)
    if result.tests_pass:
        sandbox.export_patch()    # 通过即提交
        break
    # 失败的 stderr 进入下一轮 obs
```

**示例：OpenHands SDK**
```python
from openhands.sdk import Agent, DockerSandbox

agent = Agent(llm="claude-sonnet-4.5")
with DockerSandbox(image="python:3.12") as sb:
    result = agent.run(
        task="Fix issue #42 in this repo",
        sandbox=sb,
        success_criteria=lambda: sb.exec("pytest").returncode == 0,
    )
```

**优点**：抓**运行时错误 + 真实功能错误**
**局限**：依赖测试覆盖率；sandbox 启动成本；非 hermetic 环境难复制

---

### ③ 测试即门控（TDD-as-verification）

**机制**：先写测试，agent 必须让测试过——把"门控"前移到生成动作之前；失败时自动喂回 LLM。

**代表**：
- **Aider 的 `--test-cmd` / `--lint-cmd`**（**最成熟的开源闭环**）
- **Codium AI / Qodo** — 自动生成测试 + 测试门控
- **CodeT** (research) — LLM 同时生成 N 候选 + N 测试，测试投票选最佳
- **AlphaCode 2** (DeepMind) — 一题生 1M 候选，编译+测试+隐藏测试三道滤网

**示例：Aider 的 test-cmd 闭环**
```bash
# Aider 启动时挂上 test cmd
aider --test-cmd "pytest tests/" --auto-test --yes

# 内部循环（伪代码）：
# 1. 用户提需求 → LLM 出 diff
# 2. apply diff
# 3. 自动跑 pytest tests/
# 4. 如果 fail → 把 pytest 输出 + diff 贴回 LLM 让它修
# 5. 循环到 pass 或者达到 retry 上限
```

**示例：CodeT 投票思路（伪代码）**
```python
candidates = [llm.generate_solution(problem) for _ in range(50)]
generated_tests = [llm.generate_test(problem) for _ in range(20)]

scores = {}
for solution in candidates:
    sandbox = Sandbox(solution)
    pass_count = sum(1 for t in generated_tests if sandbox.run_test(t).pass_)
    scores[solution] = pass_count

best_solution = max(scores, key=scores.get)
```

**优点**：唯一能抓"实现错了需求"的便宜方案
**局限**：需要测试存在；overfit 到测试用例

### ④ Hook / 拦截门控（动作级，最细粒度）

**机制**：在 agent 调用每个工具（Read/Write/Bash/Edit）之前/之后插钩子，钩子有阻断权——deny / ask（人确认）/ 替换参数。

**代表**：
- **Claude Code Hooks**（事实标准，最完整）：`PreToolUse` / `PostToolUse` / `Stop` / `SubagentStop` / `UserPromptSubmit` / `Notification` / `PreCompact`
- **Cursor Hooks**（轻量版，主要文件保存/编辑）
- **Cline / Roo Code** approval mode
- **Aider** confirmation prompts

**示例 1：Claude Code PreToolUse 拦截危险 Bash**
```json
// ~/.claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "/Users/me/.claude/hooks/guard-bash.sh"
      }]
    }]
  }
}
```

```bash
#!/usr/bin/env bash
# guard-bash.sh —— stdin 是 hook 上下文 JSON
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command')

if echo "$cmd" | grep -qE 'rm -rf|drop database|sudo'; then
  jq -n --arg reason "Dangerous command blocked: $cmd" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: $reason
    }
  }'
  exit 0
fi

if echo "$cmd" | grep -qE 'migrate|schema|alembic'; then
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "ask",
      permissionDecisionReason: "Schema-related command, please confirm"
    }
  }'
fi
```

**示例 2：Claude Code Stop hook 强制 agent 继续工作**
```bash
#!/usr/bin/env bash
# stop-guard.sh —— agent 想结束时检查测试是否真的过了
if ! pytest tests/ --quiet > /dev/null 2>&1; then
  jq -n '{
    decision: "block",
    reason: "Tests are still failing. Continue fixing them before stopping."
  }'
fi
```

**示例 3：PostToolUse 检查 Edit 是否破坏 schema**
```bash
#!/usr/bin/env bash
# post-edit-schema-check.sh
input=$(cat)
edited=$(echo "$input" | jq -r '.tool_input.file_path')

if [[ "$edited" =~ schema|migration ]]; then
  if ! python -m alembic check 2>/dev/null; then
    jq -n '{decision: "block", reason: "Schema check failed after edit. Please review."}'
  fi
fi
```

**优点**：粒度极细；能抓灾难前夜；可做策略而非检查
**局限**：跨 IDE 不可移植；Codex CLI 没原生 hook

### ⑤ AI Critic 门控（critic-actor 模式）

**机制**：一个 LLM 生成、另一个 LLM 评审，评审不通过就回炉。

**代表**：
- **CodeRabbit** — PR 级 AI 评审，可配置成 required check
- **Cursor BugBot** — Cursor 自家 PR-mode bug 评审
- **Greptile** — 深度 indexing 整 repo
- **GitHub Copilot Code Review** — GitHub 内建
- **Sourcery** — Python 重构建议
- **Qodo Merge** — 自动评审 + 测试生成
- **Self-Refine / Reflexion** (research) — 同 LLM 自评的 single-agent 退化版

**示例 1：CodeRabbit 配置**
```yaml
# .coderabbit.yaml
language: en-US
reviews:
  request_changes_workflow: true   # 评审失败 = required check 失败
  high_level_summary: true
  poem: false
  review_status: true
  collapse_walkthrough: false
  path_filters:
    - "!**/*.lock"
    - "!**/dist/**"
  path_instructions:
    - path: "app/api/**"
      instructions: "FastAPI routes; check Pydantic model validation."
    - path: "**/migrations/**"
      instructions: "Migration files; verify reversibility & data integrity."
chat:
  auto_reply: true
```

**示例 2：自定义 critic agent 提示词**
```markdown
# pl/agents/pl-critic.md
---
name: pl-critic
description: 评审 D gate 之后的 patch，输出阻断/通过决定 + 评论列表
allowed_tools: [Read, Bash]
---

You are a code review agent. You receive:
- A diff (via stdin)
- The change spec (path: pl/changes/<id>/spec.md)
- The trace of D gate checks (path: pipeline-output/trace/<id>.events.jsonl)

Output JSON:
{
  "decision": "approve" | "request_changes" | "block",
  "comments": [
    {"file": "...", "line": ..., "severity": "error|warning|info", "message": "..."}
  ],
  "summary": "..."
}

Criteria:
- Logic errors, off-by-one, null handling, race conditions
- Naming consistency with codebase
- Adherence to spec.md feature list (don't approve scope creep)
- Test coverage of new code paths
```

**优点**：抓**语义错误**能力强；可编码组织规范；可解释性高
**局限**：成本（每次 LLM 调用）；幻觉；adversarial（coder 可学会骗 critic）

### ⑥ 多 Agent SOP 门控（流程化分工，自带门控）

**机制**：把开发拆成预定义的角色和阶段，每个阶段有"产出标准"才能传给下一角色。

**代表**：
- **MetaGPT** — `Code = SOP(Team)`：PM → Architect → Project Manager → Engineer → QA
- **CrewAI** — `Process.hierarchical` + manager delegate
- **AutoGen** — `GroupChat` 多 agent 互查
- **LangGraph supervisor pattern** — supervisor 路由 + critic 子图
- **MS GitHub Copilot Workspace** — task→spec→plan→implementation→test
- **ChatDev** — 类 MetaGPT 但偏角色对话

**示例：MetaGPT 角色定义**
```python
from metagpt.actions import Action
from metagpt.roles import Role

class WriteTest(Action):
    async def run(self, code: str) -> str:
        return await self._aask(f"Write pytest tests for:\n{code}")

class QAEngineer(Role):
    name = "QA"
    profile = "Quality Assurance"
    goal = "Ensure all code has >80% test coverage"
    constraints = "Reject any code without tests"

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self.set_actions([WriteTest])
        self.watch_actions = [WriteCode]   # 等 Engineer 产出 code 后触发
```

**优点**：流程清晰、产物结构化、易追溯
**局限**：角色互动开销大；流程僵化；**SWE-bench 实测被 SWE-agent 单 agent + tool-use 击败**

### ⑦ 流程阶段门控（Spec / Constitution 优先）

**机制**：强制走完前置阶段才能进下阶段，每阶段有"完成的硬标准"。

**代表**：
- **GitHub Spec Kit** — `/speckit.constitution → /spec → /plan → /tasks → /implement`
- **pl-pipeline** — 当前主战场（七阶段 + 八门控）
- **OpenSpec** — pl 的前身概念
- **Cursor Plans** — 强制规划再执行
- **Devin 的 plan/exec 模式**
- **MS Plan-and-Execute** — LangGraph 经典模式

**示例 1：Spec Kit constitution 模板**
```markdown
# .specify/memory/constitution.md
## I. Library-First
Every feature begins as a standalone library; no inline implementations.

## II. CLI Interface
Every library exposes functionality via CLI: stdin/args → stdout, JSON.

## III. Test-First (NON-NEGOTIABLE)
TDD mandatory: write tests → user approves → tests fail → implement.

## IV. Integration Testing
Required for: new contracts, contract changes, inter-service comm.

## V. Observability
Structured logging required; multi-tier logs flow to unified stream.

---
**Sync Impact Report (auto-generated)**
- Modified principles: III (added "user approves")
- Modified templates: ✅ plan-template.md, ⚠ spec-template.md (manual review)
- Version: 2.1.0 → 2.2.0 (MINOR: principle expanded)
```

**示例 2：pl 的 gate 配置（已有）**
```yaml
# pl/config.yaml
gates:
  D:
    from: IMPLEMENT
    to: VERIFY
    checks: [compile, lint]
    eval: all_checks.pass
    on_failure: block
```

**优点**：强结构化；可追溯；适合复杂变更
**局限**：流程开销大；门控只看"产物存在"不看"产物正确"；易形式主义

### ⑧ 人在环门控（HITL，最朴素）

**机制**：人来批准——PR review、Cline approval mode、Cursor BugBot 评论后人决定。

**代表**：
- **GitHub Branch Protection** — required reviewers / required status checks
- **Cline / Roo Code** — 默认每个动作弹窗
- **Cursor BugBot** + 人 review 组合
- **Aider** 默认 confirmation
- **Slack/Linear** 集成审批

**示例 1：GitHub Branch Protection**
```yaml
# 通过 gh CLI 配置
gh api -X PUT repos/:owner/:repo/branches/main/protection \

```

**示例 2：Cline approval mode 行为**
```
[Cline] I want to run: rm -rf node_modules && npm install
[Cline] Approve? (y/n/edit)

[Cline] I want to write to package.json (added dep "lodash@^4.17")

```

**优点**：可靠；合规友好
**局限**：慢、贵；人会"approval fatigue"狂点 yes

## 3. pl 当前门控 vs 八大范式对照表

| 范式 | pl 现状 | 强度 | 主要缺口 |
|---|---|---|---|
| ① 静态检查 | D gate (compile + lint) via adapter 注入 cmd | ⭐⭐⭐⭐ | SAST/SCA/secret-scan 没接入 |
| ② 执行/沙盒 | 直接用户 shell 跑（无沙盒） | ⭐⭐ | 危险动作无隔离；多机不可复现 |
| ③ 测试即门控 | E gate (test) + E_smoke probe | ⭐⭐⭐ | **没有 fail→re-generate 闭环** |
| ④ Hook 拦截 | 几乎空白 | ⭐ | 不在 agent 工具调用闭环 |
| ⑤ AI critic | 完全没有 | ⭐ | 缺二号 LLM 评审 |
| ⑥ 多 agent SOP | 配置上 owner_actor，实操同一 LLM | ⭐⭐ | 没有真分工 |
| ⑦ 流程阶段 | A0/B1/C/D/E/E_smoke/F/G + artifact + eval | ⭐⭐⭐⭐⭐ | 已是头部 |
| ⑧ 人在环 | A0/B1 隐式签字 | ⭐⭐ | 无显式 approval 结构化记录 |

## 4. pl 门控系统真实优势 / 真实劣势

### 优势（应保留并发扬）

1. **配置驱动 + Schema 化**：`gate.schema.json` + `pl/config.yaml` 让门控可声明、可校验、可被外部工具理解。MetaGPT/CrewAI 都是代码内嵌，pl 是配置——可读性、可审计性、可外部消费性更强
2. **`adapter` 注入命令机制**：`${PL_BUILD_CHECK_CMD}` 让门控的"具体命令"由项目类型决定，门控逻辑跨项目复用——**社区独此一份**
3. **Trace JSONL 作为门控证据**：所有 check 结果落 trace，门控 eval 读 trace，**门控决策可事后审计** — 比 Husky/CI 这类"跑完结果丢了"强得多
4. **Stage 与 Gate 解耦**：gate 有 from/to，可灵活组装新流程；不是硬编码
5. **on_failure 三态（block/warn/skip）**：很多框架只有 block，pl 的 warn 给探索性流程留口子
6. **internal 子门控（C/F）**：在 stage 内做小循环检查，不要求换阶段——很多框架没这概念

### 劣势（亟需补强）

1. **没有动作级 hook**——pl 只在阶段切换时门控，agent 中途想 `rm -rf` 拦不住
2. **没有 fail→re-generate 闭环**——D/E gate 失败就 block，不会自动喂回 LLM。Aider 这一点完胜
3. **没有 sandbox**——直接 user shell，环境隔离差
4. **没有 AI critic**——质量门控只到 lint/test 层，逻辑/语义错误没人抓
5. **`eval` 表达力弱**——只支持枚举三选一；不能写"D 失败 → 自动调用 fixer agent"这种条件路由
6. **artifact 契约只看"存在"不看"质量"**——`spec.md` 存在就过，没人检查它是否真的写完整
7. **没有 approval token 机制**——人签字没结构化记录
8. **没有跨 change 门控聚合**——比如"本周累计 3 次 D gate 失败 → 触发流程审计"

---

## 5. 六个增强方向（按 ROI 排序，每方向附 pl 落地示例）

### 方向 A — D/E gate 加 "fail→re-generate" 闭环 ⭐⭐⭐⭐⭐ 最高 ROI

**学**：Aider `--test-cmd` 闭环
**实现**：扩 `gate.schema.json` 加 `on_failure: retry_with_agent` 模式 + `retry_max: 3` + `retry_agent: <agent_name>`

**pl 落地示例**：
```yaml
# pl/config.yaml
gates:
  D:
    from: IMPLEMENT
    to: VERIFY
    checks: [compile, lint]
    eval: all_checks.pass
    on_failure: retry_with_agent
    retry_max: 3
    retry_agent: pl-fixer
    fallback: block        # retry_max 用尽后的兜底动作

  E:
    from: VERIFY
    to: SMOKE
    checks: [test]
    eval: all_checks.pass
    on_failure: retry_with_agent
    retry_max: 5
    retry_agent: pl-fixer
    retry_context:
      include_stderr: true
      include_failed_check_logs: true
      include_recently_changed_files: true
```

```markdown
<!-- pl/agents/pl-fixer.md -->
---
name: pl-fixer
description: 接收失败 check 的 stderr 和上下文，修复代码让 check 通过
allowed_tools: [Read, Edit, Bash]
---

You are a fixer agent. Input includes:
- Failed gate name + check name(s)
- stderr / stdout from the failed check
- Recently changed files in this task
- The task spec from pl/changes/<id>/spec.md

Your job: fix the code so that the next D/E gate run passes.
- Make minimal changes
- Don't add new features
- Don't disable tests; fix the underlying issue
- If you can't fix in 3 edits, output `cannot_fix: <reason>` and stop.
```

**收益**：贴 pl 现有架构、最小 schema 改动、对体验改善巨大
**风险**：retry_max 设错会无限烧 token；需要 critic 逃避检测（用 fixer 而不是禁用 test）

---

### 方向 B — PreToolUse / PostToolUse 风格 hook ⭐⭐⭐⭐

**学**：Claude Code hooks
**实现**：在 `pl/hooks/` 下放 shell/python 脚本，由 IDE 集成（Claude Code / Cursor）触发

**pl 落地示例**：
```yaml
# pl/config.yaml
hooks:
  PreToolUse:
    - matcher: "Bash"
      script: pl/hooks/guard-bash.sh
      description: "拦截危险 Bash 命令"
    - matcher: "Edit"
      file_pattern: "**/migrations/**"
      script: pl/hooks/ask-on-migration.sh
  PostToolUse:
    - matcher: "Edit"
      file_pattern: "**/schema.py"
      script: pl/hooks/check-schema-after-edit.sh
  Stop:
    - script: pl/hooks/verify-tests-pass.sh
      description: "agent 想结束时检查测试"
```

```bash
# pl/hooks/guard-bash.sh
#!/usr/bin/env bash
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command')

source "$PL_HOME/scripts/trace-emit.sh"
trace_emit "hook.triggered" "$(jq -cn --arg cmd "$cmd" '{hook:"PreToolUse-Bash",cmd:$cmd}')"

if echo "$cmd" | grep -qE 'rm -rf /|drop database|TRUNCATE.*production'; then
  trace_emit "hook.blocked" "$(jq -cn --arg cmd "$cmd" '{hook:"PreToolUse-Bash",cmd:$cmd,reason:"dangerous"}')"
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "Dangerous command blocked by pl hook"
    }
  }'
fi
```

**配套**：`pl ide sync` 时自动把 `pl/hooks/` 写入 IDE settings（Claude Code 的 `~/.claude/settings.json` / Cursor 的 hook 配置）

**收益**：补 ④ 范式空白；可拦危险动作；与 trace 体系集成
**风险**：跨 IDE 实现路径不同；需要 manifest 抽象

---

### 方向 C — 轻量 AI critic（在 D/G gate 之间插一层） ⭐⭐⭐⭐

**学**：CodeRabbit + 自定义 critic agent
**实现**：D pass → 跑 `pl-critic` agent → 输出阻断/通过决定 + 评论

**pl 落地示例**：
```yaml
# pl/config.yaml
gates:
  D_critic:
    from: IMPLEMENT
    to: VERIFY
    type: critic
    agent: pl-critic
    on_failure: warn       # 默认 warn 不 block，避免幻觉误伤
    threshold:
      max_errors: 0
      max_warnings: 5
```

```markdown
<!-- pl/agents/pl-critic.md -->
---
name: pl-critic
description: D gate 后审 patch；输出 JSON 评论 + decision
allowed_tools: [Read, Bash]
input_format: stdin
output_format: json
---

# Inputs (via stdin)
{
  "change_id": "...",
  "diff": "...",         /* git diff */
  "spec": "...",         /* spec.md content */
  "trace_summary": "..." /* recent trace events */
}

# Output (stdout JSON)
{
  "decision": "approve" | "request_changes" | "block",
  "errors": [...],
  "warnings": [...],
  "summary": "..."
}

# Criteria
- Logic errors: off-by-one, null handling, race conditions
- Naming consistency with rest of codebase
- Spec adherence (no scope creep)
- Test coverage of new paths
- DO NOT comment on lint-fixable issues (already covered by D static gate)
```

**收益**：补 ⑤ 范式空白；抓语义错误
**风险**：成本（每次 LLM 调用）；幻觉率；初期默认 warn 不 block

---

### 方向 D — DSL 化的 `eval` 表达式 ⭐⭐⭐

**学**：Argo Workflow / Tekton conditions / OPA Rego
**实现**：用 [`expr-lang`](https://github.com/expr-lang/expr) 或 Python `simpleeval` 解析

**pl 落地示例**：
```yaml
# pl/config.yaml
gates:
  D:
    checks: [compile, lint]
    # 旧格式仍兼容
    # eval: all_checks.pass

    # 新格式：DSL
    eval: |
      checks.compile.pass &&
      (checks.lint.pass || checks.lint.warning_count < 5)

  D_advanced:
    checks: [compile, lint, security_scan]
    eval: |
      checks.compile.pass &&
      checks.lint.pass &&
      (checks.security_scan.severity != "critical" ||
       approvals.count("security-team") >= 1)
    routes:                         # 条件路由
      - when: "checks.compile.fail"
        action: invoke_agent
        agent: pl-fixer
        retry_max: 3
      - when: "checks.security_scan.severity == 'critical'"
        action: notify
        target: security-team
```

**可用变量**：
- `checks.<id>.pass | fail | duration_ms | warning_count | error_count | severity`
- `gates.<id>.result`
- `artifacts.<name>.exists | size | sha256`
- `approvals.count(<role>)`
- `retries.count`

**收益**：表达力暴涨；条件路由成为可能
**风险**：DSL 学习成本；需要严格沙盒（不能 eval 任意代码）

---

### 方向 E — Sandbox 模式（opt-in） ⭐⭐⭐

**学**：OpenHands / SWE-agent
**实现**：`pl runner --sandbox=docker`，配合 adapter 提供 `Dockerfile.dev`

**pl 落地示例**：
```yaml
# pl/config.yaml
sandbox:
  enabled: false                 # opt-in
  default_backend: docker        # docker | podman | firecracker
  image_from_adapter: true       # 用 adapter 的 Dockerfile.dev

gates:
  D:
    sandbox: true                # 此 gate 强制沙盒
    checks: [compile, lint]
```

```yaml
# adapter 侧（如 adapter-python-fastapi）
# .pl-adapter.yaml
sandbox:
  dockerfile: ./Dockerfile.dev
  workdir: /workspace
  mount_host: read-only
```

```bash
# pl-runner 沙盒模式（伪代码）
if [[ "$gate_sandbox" == "true" ]]; then
  docker run --rm \
    -v "$PL_PROJECT:/workspace:ro" \
    -v "$PL_OUTPUT/sandbox-out:/out" \
    "$adapter_image" \
    bash -c "$check_cmd" \
    > "$check_log" 2>&1
fi
```

**收益**：补 ② 范式空白；多机可复现；危险动作隔离
**风险**：Docker 依赖；启动慢；不所有场景适用 → opt-in

### 方向 F — 显式 approval token + audit ⭐⭐

**学**：GitHub branch protection required reviewers + Sigstore
**实现**：`/pl:approve <gate>` 写结构化 trace event；可选 GPG 签名

**pl 落地示例**：
```yaml
# pl/config.yaml
gates:
  A0:
    from: SPEC
    to: PLAN
    checks: [spec_complete]
    requires_approval:
      role: spec_owner       # 谁有权批准
      mode: explicit         # explicit | implicit
      ttl: 24h               # 批准多久后过期需重新批准
```

```bash
# 用户端命令
pl approve A0 --reason "spec reviewed, no blockers"
pl approve A0 --revoke

# 内部写 trace event
trace_emit "approval.granted" '{
  "gate": "A0",
  "role": "spec_owner",
  "user": "guobin@local",
  "reason": "spec reviewed, no blockers",
  "ttl_unix": 1714209900,
  "signature": "<optional GPG sig>"
}'
```

**收益**：补 ⑧ 范式空白；合规审计材料齐全
**风险**：稍微繁琐 → 可加 `--auto-approve` 个人模式开关

## 6. 反直觉洞察 + 优先级判断

### 6.1 研究界 2024-2025 实测教训

> **MetaGPT/ChatDev 这类"重 SOP+多角色"在真实 SWE-bench 上输给 SWE-agent 这类"单 agent + 强 tool-use + sandbox + 测试反馈"。**
>
> 原因：**门控密度 ≠ 质量**。多 agent 互查带来 N 倍 token 成本和 N 倍幻觉源，
> 而单 agent 拿到真实运行结果（execution observation）做反馈最有效。

### 6.2 对 pl 的启示

**ROI 排序**（不要为加门控而加门控）：

```
方向 A (retry 闭环)  ████████████████████  最高 ROI
方向 B (Hook)        ██████████████        高
方向 C (Critic)      ██████████            中高
方向 D (DSL eval)    ████████              中
方向 E (Sandbox)     ██████                中（opt-in）
方向 F (Approval)    ████                  低（可选）
完整重构成 MetaGPT     ⊘                   不推荐
```

**优先级原则**：
1. **真实运行反馈 (③) > AI 评审 (⑤) > 多 agent 编排 (⑥)**
2. **静态门控 (①, 已有) > Sandbox (②) > Hook (④)**
3. **流程阶段门控 (⑦) 已 ⭐⭐⭐⭐⭐ 不必再加**

### 6.3 一句话画像各家方案

| 项目 | 一句话画像 |
|---|---|
| **pl-pipeline** | "**最重的流程门控** + 中等静态检查；缺动作 hook、缺 critic、缺反馈闭环" |
| **Spec Kit** | "**最轻的流程门控**（4 阶段），靠 constitution 软约束" |
| **Aider** | "**测试反馈闭环之王**——edit→test→fix 自循环；无流程门控" |
| **Claude Code** | "**Hook 之王**——动作级阻断最强；无显式流程门控" |
| **MetaGPT/CrewAI** | "**SOP 化多 agent**——重，研究价值高，实战效果存疑" |
| **OpenHands/SWE-agent** | "**Sandbox 之王**——hermetic 环境 + 真实测试，benchmark 跑分领先" |
| **CodeRabbit** | "**AI critic 之王**——PR 级评审最成熟" |
| **Devin** | "**端到端封闭** sandbox + 流程 + critic 的商业整合" |

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| **retry 闭环烧 token** | 高 | 中 | `retry_max` 强制上限；trace 记录每次 retry 的成本；超阈值 alert |
| **AI critic 误伤通过的 PR** | 中 | 中 | 默认 `on_failure: warn` 不 block；2 周观察期后再决定是否升 block |
| **hook 脚本写崩 agent** | 中 | 高 | hook 必须超时（5s）；hook 错误 fail-open（默认放行 + warn）而非 fail-closed |
| **DSL eval 安全漏洞** | 低 | 高 | 严格沙盒（仅允许列表中的字段访问，无任意代码执行） |
| **Sandbox 启动慢导致流程拖沓** | 中 | 中 | 默认关闭；只在关键 gate 开启；用 image cache |
| **多 agent 重复添加** | 低 | 中 | 章节 6.1 的研究教训写入 ROADMAP "非目标" |
| **Approval token 影响个人体验** | 中 | 低 | `--auto-approve` 开关；个人模式默认隐式签字 |

## 8. 执行计划（Phase 0-N，**分方向独立可上线**）

不必整体推进，每个方向都可以**独立切版本号**。建议顺序：

### Phase A（方向 A：retry 闭环）— **v1.14.0**，0.5 天
- 扩 `gate.schema.json` 加 `on_failure: retry_with_agent` + `retry_max` + `retry_agent` + `retry_context`
- `pl-runner.sh` 实现 retry 循环
- 写 `pl/agents/pl-fixer.md` 模板（adapter 不依赖具体 LLM）
- shop_agent 验证一遍
- 文档 + CHANGELOG

### Phase B（方向 B：Hook）— **v1.14.1**，1-2 天
- 设计 `pl/hooks/` 目录结构
- 写 `pl/hooks/_template/` 几个示例（guard-bash / ask-on-migration / verify-tests-stop）
- 扩 `pl ide sync` 把 hook 注入 Claude Code / Cursor settings
- Codex CLI 因无原生 hook，提供"手工 source"指南

### Phase C（方向 C：Critic）— **v1.14.2**，1 天
- 写 `pl/agents/pl-critic.md` 模板
- gate.schema.json 加 `type: critic` + `agent` + `threshold`
- 集成到 `pl-runner.sh` D 之后调用
- 默认 warn 模式，文档说明何时升 block

### Phase D（方向 D：DSL eval）— **v1.15.0**，1 天
- 选 expr 引擎（Python `simpleeval` 是 bash 友好的轻量首选）
- 扩 `gate.schema.json` 的 `eval` 字段为 `oneOf: [enum, string]`
- 旧 `eval: all_checks.pass` 兼容
- 实现 `routes` 条件路由
- 文档：DSL 可用变量字典

### Phase E（方向 E：Sandbox，opt-in）— **v1.16.0**，2-3 天
- adapter 侧约定 `Dockerfile.dev` 契约
- `pl-runner.sh` 加 `--sandbox=docker` 模式
- 文档：哪些场景应该开

### Phase F（方向 F：Approval）— **v1.17.0**，0.5 天
- `bin/pl approve` 命令
- gate.schema.json 加 `requires_approval` + `ttl`
- trace event 类型 `approval.granted` / `approval.revoked`

**总计**：6-9 个工作日（分版本独立）。
