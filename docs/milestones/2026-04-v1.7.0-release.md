### 2.3 v1.7.0 · CDC 双向闭环（本次主题）

```
adapters/<adapter>/adapter.yaml
    ↑ provides.capabilities[]   （v1.7 新增段——稳定的对外能力承诺）
    │
   read by
    │
scripts/pl-contract-aggregate.sh  （把 adapter.use 流聚合成 pact）
    │
    └→ pl/contracts/<change>.consumed.yaml   （consumer pact，每个 change 一份）

            │

            └→ 集成 CI：consumer 项目和 adapter PR 都拿来跑
```

## 4. 关键 bug 与修复（A1 dogfood 实战收获）

### 🐛 `pl-runner.sh` 在 criteria-only gate 上绕过 ARCHIVE 自动 aggregate

**触发条件**：gate 只声明 `criteria:`（v1 兼容写法），不声明 `checks:`。

**根因**：`pl-runner.sh` 在 `ITEMS_COUNT == 0` 时早出 `exit 0`，绕过了 `maybe_run_archive_aggregate` 调用。
v1.7.0 release 前的合成 E2E 全用了带 `checks:` 的 gate，没覆盖到这个边界。

**修复**：
- commit `a5f1e26` · 抽出 `maybe_run_archive_aggregate()` 函数，正常路径和 early-exit 路径**两处**调用。
- commit `2f7a3b7`（A2 retro 同时记录跨 adapter 通用性）。

**回归保护**（A4，本次新增）：
- `.github/workflows/ci.yml` 的 `cdc-broker-smoke` job 新增 **Scenario 4**：
  显式构造 criteria-only ARCHIVE gate，断言 `auto-aggregate` 字符串出现 + pact + registry 文件存在。
- 不加这个 fixture，下次重构 pl-runner 极易 silently 回归。

## 5. 设计决策（为什么这样而不是那样）

### 5.1 为什么 broker = pure 脚本 + git，不是 HTTP 服务

| 选项 | 优势 | 劣势 |
|---|---|---|
| HTTP service（pact-broker 那套） | 跨仓库分发方便 | 需要服务运维、鉴权、备份；CI 要拉服务依赖 |
| **本方案：脚本 + 文件 + git** | 无服务依赖；pact 进 git diff 直接可审；CI 用现有 checkout 即可 | 跨仓库 consumer 必须 vendor pl-pipeline 或用 submodule |

→ 选择后者。pl-pipeline 是单人/小团队工具，"零运维"权重远高于"分布式 broker"。

### 5.2 为什么用 `provides.capabilities[]` 而不是直接绑 skill 名

A2 的 N3 实证：consumer pact 直接记 `build_command/compile_check`，adapter 想把 `compile_check` 改名 `typecheck` → 一升级就 broken。

`capabilities[]` 给 adapter 留了**演进退路**：
- consumer 签约 `capability/typecheck`
- adapter 内部 `backed_by: { build_command: <whatever-name> }` 可自由重命名
- `since/deprecated_in/removed_in` 提供软迁移期（warn → broken 渐进）

→ A2 retro 第 4.2 节有详细对照。**A3 把 fastapi/kotlin 也补齐了 capabilities**，正是为了让所有 adapter 都有这条退路。

### 5.3 为什么 ARCHIVE 的 auto-aggregate 是非阻塞的

`maybe_run_archive_aggregate` 失败时 `log_warn` 而非 `exit 1`。理由：
- 观测层不能成为流水线故障源（违反 v1.6.0 的设计原则）
- aggregate 失败时人工兜底（手跑 `pl-contract-aggregate.sh --change <id>`）
- 真正的把关在 PR review 时的 `pl-contract-verify --strict`，不在 ARCHIVE 时
