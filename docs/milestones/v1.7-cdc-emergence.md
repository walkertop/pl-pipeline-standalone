# v1.7 CDC Emergence · 为什么计划外突然冒出来一个契约系统

> **写于**：2026-04-24（事后归档，与 v1.6.x / v1.7.x tag 同期）
> **目的**：v1.6-next-planning.md 没有 CDC 这条线，但 v1.7 一日内打了 4 个 tag。
> 本文回答"为什么是 CDC、为什么是现在、为什么不是 H（retro-miner）"。
> **配套**：[`2026-04-v1.7.0-release.md`](./2026-04-v1.7.0-release.md)（功能详解）

---

## TL;DR

E' 跑完 Pomodoro Timer 真实流水线后暴露了一个**原 ROADMAP 没看到的缺口**：

> adapter 单方面声明"我提供什么" → ✅ 已有（piao 的 `ContractDrift`）
> consumer 实际用了"什么、什么时候、在哪个 phase" → ❌ 没有
> adapter 升级 / 删能力 / 改名时 → consumer **静默被打到**，CI 不拦

retro-miner（Path H）能挖"用得多 vs 用得少"的频次，但**不能拦 breaking**。
这一刻判断"事实账本"比"挖掘"紧迫，于是 H 推迟、CDC 上车。

## 2. 与原 ROADMAP 的对照

### 2.1 已有的契约/观测能力

ROADMAP 在 v0.2 列了 4 个本体抽象，3 个落地了：

- ✅ **E1 GateDefinition / CheckDefinition**（v1.1.0-alpha）
- ✅ **E2 SMOKE 阶段一等公民**（v1.1.0-alpha）
- ✅ **E3 rule-as-code**（v1.1.2-alpha）
- ✅ **E4 piao-contract-drift-compute**（v1.1.1-alpha）— **adapter→host 单向**

E4 解决的是"adapter 声称这个项目应该长什么样 vs 实际长什么样"。
**它解决不了"consumer 实际用了 adapter 什么"反向问题**。

### 2.2 ROADMAP 长期主题里有暗示，但没立项

```
长期主题（v1.x+）:
  Pipeline-as-Code
  研发"飞轮"沉淀
  跨仓 change 联动
```

"研发飞轮"提了"组件复用率达阈值时推荐 promote to shared"，**这要求先有
"谁用了什么"的账**。但 ROADMAP 没把这个账本立成单独的里程碑，所以没排期。

### 2.3 v1.6-next-planning.md 的 4 条路径

| Path | 与 CDC 关系 |
|---|---|
| E' 真实使用验证 | CDC 的**触发器**（E' 暴露问题） |
| F 扩充 adapter | CDC 的**受益方**（多 adapter 时 CDC 价值放大），但**前提**也是 CDC 先有 |
| G CLI 打包 | 与 CDC 正交（v1.8~v1.10 同期完成） |
| H retro-miner B→C→D | **CDC 的下游**：CDC 产出 pact 数据，retro-miner 才能挖出有价值的 pattern |

**所以**：CDC 不是 4 条路径的"替代"，而是它们**共同的下游基础设施**——
4 条路径都假设"我们知道 consumer 用了 adapter 什么"，但这件事原本没人做。

## 3. 为什么不是 Path H？

### 3.1 H 的设计是

> "retro-miner 吃真实 trace → 挖 4 类 pattern → 反推契约规则"

### 3.2 但 E' 跑完后真实情况是

- trace 总量 ~150 行 jsonl（一个小 demo）
- adapter 用了几次？**trace 里没记**（因为 `trace-adapter-use.sh` v1.6.1 才有）
- 跨 change 的频次共现？**只有 1 个 change，没法共现**

**结论**：retro-miner 在 trace 没有 `adapter.use` 事件、没有跨 change 数据
之前跑，挖出来的全是"环境噪音"。**先把账本建起来，再让挖掘有饭吃**。

### 3.3 推迟 H 的代价

- 飞轮闭环延后到 v1.11+（等 CDC 产生足够多 pact 数据后再上 retro-miner）
- 短期看不到"挖出 pattern 反哺 rules"的能力

### 3.4 不推迟 H 的代价（如果硬上）

- retro-miner 输出空报告 → 用户失望 → 信心受损
- 输出错误 pattern（噪音被当信号）→ 反哺到 rules → 污染知识库

**净判断**：推迟 H 是正确的。

## 4. 为什么是 CDC（而不是别的"账本"形式）？

### 4.1 候选方案对比

| 方案 | 优点 | 缺点 | 选 |
|---|---|---|---|
| A. Adapter usage log（一个全局 jsonl） | 实现最快 | 没有 schema、没法 diff、CI 难拦截 | ❌ |
| B. 每个 change 一份 `consumed.yaml`（per-change pact） | 与 trace 同语义、可 commit、可 diff | 跨 change 反查需聚合 | ✅ 主形态 |
| C. **CDC 双向**：B + adapter 侧 `provides.capabilities[]` + verify 引擎 | 能在 adapter PR 上拦 breaking | 多一个 adapter 侧抽象层 | ✅ 最终方案 |
| D. 直接借现成 Pact（pact.io） | 生态成熟 | 引入 jvm/node 依赖、过度工程 | ❌ |

**选 C** 因为：
- B 的 per-change pact 让 consumer 侧的"事实"可读、可 review、可 git diff
- C 的 `provides.capabilities[]` 让 adapter 不暴露内部 skill/rule 名（避免脆弱耦合）
- verify 引擎让"删能力 / 改名 / 升 major" 在 adapter PR 里被拦
- 全用 bash + python3 stdlib，不引入新依赖

### 4.2 为什么"双向"

单向（只记 consumer 用了什么）解决不了：
- "我想砍这个 capability，谁会被打到？" → 需要反向索引（v1.7.1 `pl-contract-query`）
- "adapter 升级会影响哪些 change？" → 需要 verify 拿 adapter.yaml 对账所有 pact（v1.7.0）

双向闭环让 adapter 作者**升级前 30 秒 CI 就告诉你影响面**，而不是发版后等用户报 bug。

## 5. 与 v2.0.0-alpha 教训的关系

CDC 主线（v1.6.0~v1.7.3.1）和 Node.js CLI 实验（`cli/` 目录）**同期**进行：
- CDC：在主线，纯 bash + python3
- cli/：在仓根（当时），是 Node.js 平行实验

后来 v2.0.0-alpha tag 误打在 cli/ 提交上，触发了 install.sh 选错 tag 的事故。
**这两条线不应该被同等对待**：

| | CDC 主线 | Node.js CLI 实验 |
|---|---|---|
| 是否在 ROADMAP / 立项文档？ | 事后立项（本文） | 无 |
| 是否符合"零依赖"非目标？ | ✅ 是 | ❌ 引入 node |
| 是否进入 install.sh？ | ✅ | ❌ 不应该 |
| 是否打 pl-vX tag？ | ✅ | ❌ 不应该（已撤 v2.0.0-alpha） |

CDC 是"E' 反馈驱动的合理扩展"；
Node.js CLI 是"未与方向同步的私房菜实验"。
两者都在 4-23/24 提交，但只有前者应该进主线，后者应该进 `experiments/`。

## 6. 经验留给后续

1. **真实跑完一次（E'）比写 10 份 ROADMAP 更能暴露真实需求**。
   v1.6 / v1.7 都是 E' 撬出来的，不是事先想到的。

2. **"账本"先于"挖掘"**。retro-miner 在没有事实账本之前没饭吃。

3. **方向偏离需要事后归档**。本文 + `v1.6-next-planning.md` 顶部的"事后追加"
   段落让"为什么改道"留痕，避免下次 retro 失忆。

4. **实验性代码不能伪装成主线**。Node.js CLI 实验放仓根 + 打 v2.0.0-alpha tag
   = 给 install.sh 埋雷。**所有平行实现必须进 `experiments/` + 不打 pl-vX tag**。
   规则已写进 [`experiments/README.md`](../../experiments/README.md) 和 ROADMAP "非目标"。
