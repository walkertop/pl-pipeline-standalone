# Milestone · v1.5 Core Assets Migration

**状态**：📝 规划中（未开工）
**提案时间**：2026-04-23
**前置**：`pl-v1.3.2-alpha.2`（当前 HEAD）
**预期发布**：`pl-v1.5.0`（正式版，不再走 alpha）

---

## 为什么要做这件事

### 现状问题

v1.3.x 系列把观察层做完了，但 **`pl-pipeline-standalone` 目前"能用但没脑"**：

| 层面 | 现状 | 问题 |
|------|------|------|
| **流程** | ✅ pl/ 骨架 + 7 产物契约 + 6 阶段 + 7 门禁 | 没问题 |
| **契约** | ✅ schemas / templates / observe-rules | 没问题 |
| **观察** | ✅ Dashboard + SSE + retro-miner | 没问题 |
| **脑** | ❌ **`assets/pl/` 下无 skills / rules / agents** | 严重缺 |

也就是说，当一个新用户按 `dashboard-guide.md` / `working-with-fuzzy-intent.md` 想跑起来时，他会发现：
- `/pl:proposal` 命令在，但**背后没有 spec-normalizer skill 支撑**
- `/pl:archive` 命令在，但**背后没有 knowledge-archiver agent**
- 没有 `acceptance-criteria-rules` 可读
- 没有 `piao-pipeline-discipline` 强制执行
- 命令就是个空壳 `.md`，执行它时 AI 只能靠临场发挥

### 资产流向错误

所有通用能力**目前物理上住在 KuiklyPolyCity 项目里**，和 Kuikly 迁移业务深度耦合：

```
KuiklyPolyCity/.codebuddy/
├── skills/spec-normalizer/          ← 通用，但里面写满了 Kuikly 示例
├── skills/finalization-template/    ← 通用
├── rules/acceptance-criteria-rules/ ← 通用
├── rules/piao-pipeline-discipline/  ← pl/piao 本体纪律
├── agents/pipeline-master/          ← pl 编排本体
├── agents/knowledge-archiver/       ← pl archive 阶段本体
└── ...
```

这些**应该住在独立仓**，而目前只有 Kuikly 项目能用。

### 触发这个 milestone 的事件

2026-04-23 用户在审查 `working-with-fuzzy-intent.md` 的讨论中发现：

> "spec-normalizer 并没有位于 pl-pipeline-standalone，而是处于 KuiklyPolyCity 中，且和原本的 PolyCity 高度耦合。"

事实核查确认：**全仓 `grep -r spec-normalizer` 在独立仓里 0 命中**。

## 目标（本 milestone 完成后的期望状态）

### 成功标准

1. **独立仓有完整的通用资产**：`assets/pl/skills/`、`assets/pl/rules/`、`assets/pl/agents/` 目录存在并填充
2. **7 个关键资产脱敏迁移**（清单见下）
3. **KuiklyPolyCity 对应位置改为引用**（从**副本**变成**覆盖层**）
4. **任何新用户能用**：clone pl-pipeline-standalone → 照 `working-with-fuzzy-intent.md` 跑 → 所有 `/pl:*` 命令有脑

### 明确的非目标（Out of Scope）

- ❌ 不迁 Kuikly 专属资产（kuikly-debugger、weex-migration-analyzer 等）
- ❌ 不做 v0.2 CLI 打包（`pl init` 放 v2.x）
- ❌ 不重写资产内容（只做脱敏 + 搬家 + 引用关系）
- ❌ 不改 pl-core 脚本

## 脱敏规则（通用 → 独立仓时必须遵守）

### 硬规则（MUST）

1. **路径示例必须用占位符**：`shared/src/.../kuiklypolycity/...` → `<project>/<module>/<subpath>`
2. **业务词汇必须抽象**：`Kuikly 迁移` → `cross-platform refactor`；`道聚城` → `<your app>`
3. **技术栈不能钦定**：不写"用 Kotlin Flow"，改写"用你技术栈对应的响应式原语"
4. **示例不能单一**：至少给 2~3 种场景（web / mobile / CLI）

### 软规则（SHOULD）

1. 保留一份 `examples/` 子目录，允许放**具体项目的脱敏过版本**（脱掉项目名、路径、业务词）
2. 在 skill 顶部 frontmatter 加 `scope: generic | stack-specific | project-specific`，便于未来过滤
3. 脱敏后在 KuiklyPolyCity 里保留一份**"override 版"**（继承独立仓 + 加 Kuikly 特化示例）——这是"覆盖层"模式

### KuiklyPolyCity 的"覆盖层"最终形态

```
KuiklyPolyCity/.codebuddy/
├── skills/
│   ├── spec-normalizer.override.md    ← 指向独立仓 + 加 Kuikly 例子
│   ├── kuikly-debugger/               ← 项目专属，不迁
│   └── ...
└── ...
```

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 脱敏过度导致 skill 变空洞 | 保留 `examples/` 子目录放具体脱敏版示例，不是抽象空话 |
| KuiklyPolyCity 现有流程被破坏 | v1.5 发布前必须通过 override 模式回测一次 Kuikly 迁移 change |
| 脱敏质量参差 | 每份迁移资产都要生成 "Before/After Kuikly 词汇扫描报告" |
| 迁移过程 rule 版本号混乱 | 遵守 v1.3.0 已建立的 id+version frontmatter 规范；version 归零 1.0.0 |

---

## 不做这件事的后果

如果 v1.5 不做，pl-pipeline-standalone 将长期处于：

- 对外：**"空壳工具"**——新用户看 README 以为能用，真跑起来发现 `/pl:*` 命令背后没脑
- 对内：**"KuiklyPolyCity 绑架"**——所有通用改进都会先落在 Kuikly 项目，再考虑是否回迁
- 对 E' / B / C / D 后续计划：**"无米之炊"**——新项目场景跑不起来，真实 trace 产不出来，retro-miner 没饭吃
